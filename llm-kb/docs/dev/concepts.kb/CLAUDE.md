# Concepts

Structured definitions of *things* that recur across procedures, failure modes,
and case-studies. Distinct from `../glossary.kb/` (one-line word-to-meaning);
concepts carry structure: shape, authority, relationships, distinguishing
features.

## What belongs here

- One file per concept. A concept is a "thing" with properties, not a "word"
  with a meaning.
- Stub allowed: one-line meaning plus `TODO: expand`. A stub is strictly better
  than leaving a novelty undefined.
- Sections (suggested): shape, authority (when relevant), filename convention
  (when relevant), distinguished-from, see-also.

## What does NOT belong here

- One-line term-to-meaning entries → `../glossary.kb/`
- Procedures → `../procedures.kb/`
- Failure modes → `../failure-modes.kb/`
- Principles → `../principles.kb/`

## When to create a stub

When you first mention a novel term in any content file and no concept file
exists for it, create a stub. The stub is a forward-reference; it doesn't need
to be complete to be useful as a link target.
